# Community Launch Kit

Use this when inviting the first contributors. Keep the tone honest: this repo is for beginners, learning, and real contributions.

## Short LinkedIn Post

```md
I just launched Open Source Starter Lab.

It is a beginner-friendly repo for people who want to make their first real GitHub contribution without feeling lost.

The repo includes:
- good first issues
- Git and GitHub guides
- PR templates
- CI checks
- Discussions for beginner questions
- a small TypeScript CLI for contribution checklists

If you are learning open source, pick one small issue and comment that you want to work on it.
If you already know GitHub, help answer beginner questions in Discussions.

Repo: https://github.com/P-r-e-m-i-u-m/open-source-starter-lab

#opensource #github #git #typescript #beginners #goodfirstissue #developercommunity
```

## DM Invite

```md
Hey, I started a beginner-friendly open-source repo where people can practice GitHub issues, PRs, docs, and CI.

There are small good-first issues with clear steps, so it should be easy to start.

If you want, pick one issue here:
https://github.com/P-r-e-m-i-u-m/open-source-starter-lab/issues
```

## Maintainer Reply When Someone Comments

```md
Thanks for picking this up. I assigned it to you.

Please keep the PR small and include the result of `npm run check`.
If you get stuck, comment here with the command/output and I will help.
```

## First Week Plan

Day 1:
- Share the LinkedIn post.
- Invite 3-5 friends or beginner developers.
- Ask people to pick only one issue each.

Day 2:
- Reply quickly to every comment.
- Assign issues manually.
- Keep feedback simple and kind.

Day 3:
- Review first PRs.
- Ask for small improvements instead of rewriting their work.

Day 4:
- Create one Discussion question from a real beginner problem.
- Answer it with commands and explanation.

Day 5:
- Merge completed PRs.
- Thank contributors in the PR comments.

## What Not To Do

- Do not ask people to make fake PRs.
- Do not ask people to accept answers only for badges.
- Do not create spam issues.
- Do not merge broken changes just for activity.

Real activity looks better because it is better.
